# Agent Workbench 技术文档总览

## 1. 项目定位

Agent Workbench 是一个桌面端多智能体工作台。用户输入一个任务后，由总智能体拆解任务，多个子智能体并行或按依赖执行，最终由总智能体汇总结果。

本项目的核心差异化不是“返回一个 AI 答案”，而是：

> 让用户实时、显性、可追踪地看到每个智能体的执行轨迹和操作过程。

因此，开发优先级是：

```text
事件流 > 动态工作台 > 并行泳道 > Agent 卡片 > Trace 日志 > LLM 能力增强
```

第一版必须先把“看见智能体在工作”的体验做出来，再逐步增强真实 Agent 能力。

## 3. 推荐技术栈

MVP 推荐：

```text
桌面壳：Tauri 2
前端：React + TypeScript + Vite
状态管理：Zustand
任务图：React Flow，P1 阶段加入
动画：CSS Animation，必要时加 Framer Motion
本地数据：SQLite
本地后端：Node.js TypeScript sidecar 或 Tauri command 层
实时通信：WebSocket 或 SSE
LLM：Claude API 优先，ProviderAdapter 支持后续扩展
API Key：系统 Keychain / Credential Manager，不明文存 SQLite
```

### 为什么不是先做完整后端？

因为本项目第一风险不是模型能力，而是产品体验是否成立。第一阶段应先用 mock event 做出完整动态工作台，验证：

```text
用户能否看懂任务拆解？
用户能否看懂多个智能体并行执行？
用户能否从界面知道当前卡在哪里？
用户是否觉得比普通聊天框更可信？
```

## 5. 核心开发原则

### 5.1 所有行为必须事件化

任何 Agent 行为都要发事件：

```text
session 创建
任务规划
任务创建
Agent 启动
Agent 状态变化
模型输出流
工具调用开始
工具调用结束
任务完成
任务失败
需要用户输入
最终汇总
```

前端所有动态画面都从事件推导，不直接依赖后端临时状态。

---

### 5.2 UI 状态由事件流推导

不要让前端到处手动改状态。

推荐模式：

```text
events[] -> reducer -> currentWorkbenchState -> UI
```

好处：

```text
实时显示简单
历史回放简单
调试简单
失败定位简单
未来做 replay 简单
```

---

### 5.3 LLM 负责认知，程序负责秩序

总智能体负责规划和汇总，但不要让模型控制全部流程。

推荐分工：

```text
LLM：理解任务、拆解任务、执行具体认知工作、总结结果
程序：任务调度、依赖判断、状态管理、权限控制、事件记录、失败恢复
```

这样系统更稳定、可解释、可调试。

---

### 5.4 API Key 是 Provider 资源，不是 Agent 身份

Agent 配置应该是：

```text
Agent = role + systemPrompt + model + tools + budget + apiKeyRef
```

不要把“一个 Agent = 一个 API Key”写死。可以支持每个 Agent 引用不同 key，但 key 是资源配置，不是 Agent 的本质。

## 6. 标准开发流程

建议开发顺序：

```text
阶段 1：前端 Mock 动态原型
阶段 2：事件流、本地持久化与刷新恢复
阶段 3：接入 Claude API
阶段 4：任务 DAG 调度与多 Agent 并行
阶段 5：工具调用可视化
阶段 6：打磨、测试、封装桌面包
```

每个阶段的详细任务见 `11_开发计划与交接清单.md`。

## 7. 交付标准

一个开发者拿到这些文档后，应该可以明确知道：

```text
产品要做什么
第一版不做什么
采用什么技术栈
前端怎么拆组件
后端怎么拆模块
事件协议是什么
数据库怎么建表
Agent 怎么配置
任务怎么调度
LLM 怎么接入
工具怎么接入
权限怎么控制
如何测试验收
先做哪些任务
```

如果开发中遇到歧义，优先遵守本 README 中的核心原则：

> 先保证所有智能体执行过程可见，再增强智能体能力。
