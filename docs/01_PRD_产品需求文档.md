# Agent Workbench PRD 产品需求文档

## 1. 背景

现有 AI 应用多数以聊天框为中心，用户输入任务后等待最终回答。对于复杂任务，这种模式有几个问题：

```text
用户不知道 AI 是否真的在工作
用户不知道任务被如何拆解
用户不知道哪个环节失败或卡住
用户无法观察多个 Agent 的协作过程
用户难以信任最终结果
```

Agent Workbench 要解决的问题是：

> 将复杂 AI 任务从“黑箱聊天”变成“可观察、可追踪、可协作的智能体工作台”。

---

## 2. 产品目标

### 2.1 核心目标

用户输入一个任务后，系统能够：

```text
1. 由总智能体理解和拆解任务
2. 创建多个子任务
3. 分配给不同智能体
4. 支持可并行任务同时执行
5. 实时显示每个智能体的状态和执行轨迹
6. 记录每次模型输出和工具调用
7. 在所有任务完成后汇总最终结果
8. 允许用户查看过程、定位问题、复盘执行
```

---

### 2.2 产品体验目标

用户应该能一眼看懂：

```text
现在有哪些智能体
每个智能体负责什么
哪些智能体正在工作
哪些智能体在等待
哪些任务已经完成
哪个智能体调用了什么工具
哪个环节失败了
最终答案来自哪些中间结果
```

---

## 3. 目标用户

### 3.1 个人高级用户

```text
需要调研、写作、代码、资料整理等复杂任务
希望 AI 不只是聊天，而是像一个可见团队一样工作
希望过程可审查，而不是只得到一个最终结果
```

### 3.2 开发者/技术用户

```text
希望观察 Agent 的执行轨迹
希望调试工具调用和模型输出
希望看到 trace、event、token、cost
```

### 3.3 内容生产者

```text
需要多个 AI 角色共同完成选题、资料、写作、改稿、审查
希望看到每个环节的中间产物
```

---

## 4. MVP 用户场景

### 场景 1：资料调研与报告生成

用户输入：

```text
帮我调研 AI 桌面智能体的发展趋势，并输出一份分析报告。
```

系统行为：

```text
总智能体：拆解任务
研究智能体：收集资料
大纲智能体：设计结构
写作智能体：撰写正文
审查智能体：检查事实和逻辑
总智能体：最终汇总
```

用户看到：

```text
研究智能体和大纲智能体并行启动
写作智能体等待上游结果
审查智能体最后启动
所有事件在 Trace Console 中持续追加
```

---

### 场景 2：公众号文章生成

用户输入：

```text
帮我写一篇关于 AI 工具如何改变普通人工作的公众号文章。
```

系统行为：

```text
选题智能体：提炼角度
资料智能体：找案例
大纲智能体：设计结构
写作智能体：生成正文
审查智能体：润色和校验
标题智能体：生成标题
```

MVP 可以先用固定 Agent 团队模拟，不需要专门创建标题智能体。

---

### 场景 3：代码任务

用户输入：

```text
帮我做一个简单的网页 Demo。
```

系统行为：

```text
需求智能体：理解需求
设计智能体：规划 UI
编码智能体：写代码
测试智能体：运行检查
总结智能体：返回说明
```

P2 阶段加入终端、文件 diff、测试结果显示。

---

## 5. MVP 功能范围

### 5.1 顶部任务栏

功能：

```text
任务输入
选择默认 Agent 团队
开始执行
暂停任务
终止任务
查看最终结果
导出 Trace
```

MVP 要求：

```text
开始执行后输入框锁定
任务进行中显示运行状态
完成后显示最终结果入口
```

---

### 5.2 左侧 Agent 队伍

固定显示 5 个 Agent：

```text
OrchestratorAgent：总智能体
ResearchAgent：资料研究
PlannerAgent：结构规划
WriterAgent：内容生成
ReviewAgent：审查校验
```

每个 Agent 卡片显示：

```text
名称
角色
状态
当前动作
最新输出
耗时
token/cost，P1
```

状态包括：

```text
idle
planning
waiting
running
tool_call
success
failed
paused
needs_user，P1
```

---

### 5.3 中间动态工作台

MVP 优先实现并行泳道视图。

#### 并行泳道视图

```text
每个 Agent 一条横向泳道
事件块按时间出现
运行中事件块带呼吸动画
工具调用事件块带高亮动画
完成事件块变绿色
失败事件块变红色
```

用户能看到：

```text
ResearchAgent 和 PlannerAgent 是否并行
WriterAgent 什么时候开始
ReviewAgent 等待哪个任务
总智能体什么时候汇总
```

#### 任务图视图

P1 实现，展示任务依赖 DAG。

#### 回放视图

P2 实现，基于 events 表重放历史过程。

---

### 5.4 右侧详情面板

点击不同对象显示不同详情。

MVP 支持：

```text
Agent 详情
Task 详情
Event 详情
Tool Call 详情
```

内容包括：

```text
当前状态
任务说明
模型/工具信息
输入输出 payload
错误信息
最新中间产物
```

---

### 5.5 底部 Trace Console

Trace Console 显示完整事件日志。

每条日志包含：

```text
相对时间
Agent
Task
事件类型
摘要
可展开 payload
```

MVP 要求：

```text
自动滚动到底部
点击日志行后右侧展示 payload
任务完成后日志完整保留
```

## 6. 非目标

MVP 不做：

```text

云同步
插件市场

Agent 自主创建新 Agent
长期记忆系统
移动端
企业权限体系
```

## 7. 成功指标

### 7.1 体验指标

```text
用户点击开始后 2 秒内看到明显动态
至少 2 个 Agent 能并行显示
用户能通过界面说清楚谁在工作、谁在等待、谁完成了
用户点击任一事件能看到对应详情
用户能从失败状态定位失败 Agent 和失败原因
```

### 7.2 技术指标

```text
前端事件显示延迟 < 300ms
事件日志完整率 100%
同一 session 可从 events 重建 UI 状态
任务完成后可查看所有关键步骤
API Key 不明文存储
```

### 7.3 MVP 验收用例

```text
1. 打开应用，显示固定 Agent 团队
2. 输入任务，点击开始
3. 总智能体进入 planning 状态
4. 生成任务计划
5. ResearchAgent 与 PlannerAgent 并行启动
6. ResearchAgent 产生 tool_call_started 事件
7. 右侧能查看工具输入输出
8. WriterAgent 等上游完成后启动
9. ReviewAgent 最后启动
10. 总智能体输出最终汇总
11. Trace Console 记录完整过程
12. 刷新或重进 session 后能从事件恢复状态
```

## 9. 产品原则

### 原则 1：动态不是装饰，必须来自真实事件

不要做纯装饰 loading。任何动画都要对应真实执行事件。

### 原则 2：用户永远知道系统在做什么

如果任务耗时超过 3 秒，必须有可见状态更新或中间结果。

### 原则 3：复杂能力先可视化，再自动化

先让用户能看见，再让 Agent 做更多。

### 原则 4：失败必须可解释

失败时至少显示：

```text
哪个 Agent 失败
哪个任务失败
失败事件 payload
错误摘要
推荐操作
```

## 10. 第一版交互路径

```text
启动应用
↓
用户输入任务
↓
点击开始
↓
创建 session
↓
总智能体规划任务
↓
生成任务 DAG
↓
并行启动无依赖任务
↓
事件流驱动 UI 动态更新
↓
上游完成后启动下游任务
↓
全部任务完成
↓
总智能体汇总
↓
展示最终结果和完整 trace
```
