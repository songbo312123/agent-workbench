# Agent Workbench 开发计划与交接清单

## 1. 总体开发策略

本项目必须先验证核心体验：

> 用户能否动态看见多个智能体并行工作。

因此开发顺序不是先做完整 AI 后端，而是：

```text
1. Mock 动态工作台
2. 事件流架构
3. 后端调度
4. LLM 接入
5. 工具可视化
6. 权限和增强能力
```

## 11.1 P0 完成定义

P0 交付不是“页面能动”，而是核心闭环可验证：

```text
[ ] Mock 模式完整展示规划、并行、工具、写作、审查、汇总
[ ] 真实模式完整完成 session 创建、规划、调度、LLM 流式输出、最终汇总
[ ] UI 状态只从 events reducer 推导
[ ] events 可持久化并在刷新后恢复当前 UI
[ ] SSE 断线后可补齐事件
[ ] 所有 task/session 状态变更有对应事件
[ ] 关键 payload 有 shared schema 和测试
[ ] API Key 只存在系统 Keychain 或运行时内存，不进前端和 SQLite 明文字段
```

明确不属于 P0：

```text
真实联网搜索
真实文件读写
真实终端
浏览器自动化
任务图
回放
失败重试
自定义 Agent 团队
```

## 13. 开发交接清单

接手开发者需要确认：

```text
[ ] 已理解产品核心不是聊天，而是动态 Agent 工作台
[ ] 已理解事件流是系统核心
[ ] 已理解 UI 状态由 events reducer 推导
[ ] 已理解 API Key 不能进前端和 SQLite 明文字段
[ ] 已理解 P0 先做 Mock Event Player
[ ] 已理解 Orchestrator 只规划和汇总，不直接控制所有流程
[ ] 已理解 Scheduler 负责程序化调度
[ ] 已理解 ToolRuntime 是所有工具调用入口
[ ] 已理解 MVP 不做浏览器/终端/文件 diff
```

## 15. 最终提醒

不要一开始陷入模型能力、工具生态、插件系统。

第一版只要证明一件事：

> 用户能实时看见 AI 团队如何分工、并行、执行、失败和汇总。

只要这个体验成立，后面的真实工具、浏览器、终端、文件操作都可以逐步叠加。
