# Agent Workbench 工程规范与测试验收

## 1. 工程目标

工程实现必须保证：

```text
可维护
可测试
可扩展
事件完整
安全可控
前后端协议稳定
```

---

## 2. 代码规范

### 2.1 TypeScript

```text
开启 strict
不使用 any，必要时用 unknown 后校验
公共协议类型放 shared/types
事件 payload 使用 schema 校验
不要在业务代码中硬编码模型 ID、API Key、路径
```

### 2.2 命名

```text
文件：kebab-case 或 PascalCase，保持项目统一
类型：PascalCase
函数：camelCase
事件类型：snake_case
数据库字段：snake_case
```

### 2.3 错误处理

```text
所有 Provider 错误转 ProviderError
所有 Tool 错误转 ToolError
所有 API 错误返回统一格式
所有失败必须 emit 对应事件
```

## 4. 测试分层

MVP 推荐测试命令：

```text
npm run typecheck
npm run lint
npm run test
npm run test:integration
npm run test:e2e
```

如果项目脚手架采用不同包管理器，可以替换命令名，但必须保留同等检查项。

### 4.1 单元测试

必须覆盖：

```text
event reducer
plan schema validation
DAG cycle detection
getRunnableTasks
EventBus sequence generation
ToolRuntime permission check
Provider error normalization
```

---

### 4.2 集成测试

必须覆盖：

```text
POST /sessions 创建 session 和 session_started event
Orchestrator 生成任务并写入 tasks/events
Scheduler 并行运行无依赖任务
AgentRuntime 产生 message delta
ToolRuntime 产生 started/finished 事件
session 完成后写 final_answer
```

---

### 4.3 前端组件测试

建议覆盖：

```text
AgentCard 不同状态样式
LaneView 按 agent 分组事件
TraceConsole 点击事件选中 Inspector
Inspector 显示 event payload
hydrateFromEvents 从历史事件恢复 UI
```

---

### 4.4 E2E 测试

MVP 至少覆盖：

```text
打开应用
输入任务
点击开始
看到总智能体 planning
看到两个 Agent 并行启动
点击工具事件看到 payload
最终 session completed
Trace Console 有完整事件
刷新或重进 session 后状态可恢复
```

## 7. Scheduler 测试用例

### 7.1 并行任务

输入：

```text
A 无依赖
B 无依赖
C dependsOn A,B
```

期望：

```text
A 和 B 并行启动
C 等 A、B success 后启动
```

---

### 7.2 循环依赖

输入：

```text
A dependsOn B
B dependsOn A
```

期望：

```text
plan_validation_failed
session_failed
```

---

### 7.3 上游失败

输入：

```text
A failed
B dependsOn A
```

期望：

```text
B 不启动
session_failed
Trace 中显示失败原因
```

## 8. 安全测试

必须检查：

```text
API Key 不出现在 events.payload_json
API Key 不出现在前端网络响应
Agent 不能调用未授权工具
高风险工具默认被拦截
file_read 不能读 workspace 外路径
terminal 工具默认关闭
```

---

## 9. 性能目标

MVP 指标：

```text
应用启动 < 3 秒
点击开始后 2 秒内出现第一个动态事件
事件推送到 UI 延迟 < 300ms
单 session 1000 条事件内 UI 不明显卡顿
Trace Console 自动滚动不卡顿
```

P1 优化：

```text
Trace Console 虚拟列表
事件分页加载
大 payload 延迟加载
```

## 11. 验收清单

### 11.0 P0 Definition of Done

P0 完成必须同时满足：

```text
[ ] 不配置 API Key 时，Mock Event Player 可完整跑通核心流程
[ ] 配置 API Key 后，真实 session 可完成规划、调度、子 Agent 文本执行和最终汇总
[ ] 所有用户可见动态来自 events，不来自临时 UI 假状态
[ ] 所有核心事件 payload 通过 shared schema 校验
[ ] events 写入 SQLite，刷新或重进 session 后可恢复 UI
[ ] SSE 断线后可通过 afterSequence 补齐缺失事件
[ ] tasks/sessions 状态变更与对应 event 在同一事务中完成
[ ] ResearchAgent 的 P0 `web_search` 使用 mock 实现，工具事件可见
[ ] API Key 不进入前端响应、events payload、日志或 SQLite 明文字段
[ ] 单元、集成、E2E、Mock Demo、打包启动检查通过
```

```text
真实联网搜索
任务图视图
回放模式
失败重试
自定义 Agent 团队

```

## 12. 发布前检查

```text
[ ] 无 TypeScript 错误
[ ] 无 lint 阻断错误
[ ] 单元测试通过
[ ] 集成测试通过
[ ] E2E 核心路径通过
[ ] Mock Demo 可完整跑通
[ ] 无 API Key 泄露
[ ] 数据库迁移可从空库执行
[ ] 打包桌面应用可启动
[ ] README 说明清楚配置方式
```

发布阻断条件：

```text
1. 任一核心事件无法通过 schema 校验。
2. 任一 API Key、Authorization、Cookie、私钥进入 events、日志或前端响应。
3. 刷新后无法从 events 恢复当前 session UI。
4. Scheduler 可运行存在循环依赖或缺失依赖的 plan。
5. 工具调用绕过 ToolRuntime 或 Agent 白名单。
6. 打包后的桌面应用无法启动或无法完成 Mock Demo。
```
