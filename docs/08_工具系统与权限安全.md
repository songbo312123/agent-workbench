# Agent Workbench 工具系统与权限安全

## 1. 设计目标

工具系统让 Agent 能做模型本身做不了的事，例如搜索、读取文件、写文件、运行命令、操作浏览器。

但工具系统也是风险最大的部分，因此必须有清晰权限边界。

目标：

```text
1. 工具统一注册和调用
2. 每个 Agent 只能调用被授权工具
3. 工具调用全程事件化
4. 高风险工具必须用户确认
5. API Key 安全存储
6. 本地文件和终端权限可控
```

---

## 2. 工具调用架构

```text
AgentRuntime
  ↓
ProviderAdapter receives tool_use
  ↓
ToolRuntime.call(toolName, input)
  ↓
PermissionManager.check
  ↓
Tool implementation
  ↓
EventBus emit started/finished/failed
  ↓
Tool result returned to AgentRuntime
```

---

## 3. Tool 定义

```ts
type ToolDefinition = {
  name: string;
  description: string;
  inputSchema: Record<string, unknown>;
  riskLevel: "low" | "medium" | "high";
  requiresApproval: boolean;
  handler: (input: unknown, context: ToolContext) => Promise<ToolResult>;
};

type ToolContext = {
  sessionId: string;
  taskId: string;
  agentId: string;
  workspacePath?: string;
};

type ToolResult = {
  ok: boolean;
  summary: string;
  output?: unknown;
  error?: string;
};
```

---

## 4. ToolRegistry

职责：

```text
注册工具
按名称查找工具
输出工具 schema 给 ProviderAdapter
按 Agent 工具白名单过滤
```

接口：

```ts
interface ToolRegistry {
  register(tool: ToolDefinition): void;
  get(name: string): ToolDefinition | undefined;
  getToolSchemas(toolNames: string[]): ToolSchema[];
}
```

---

## 5. ToolRuntime

职责：

```text
校验工具存在
校验 Agent 是否有权限
校验 input schema
必要时请求用户授权
emit tool_call_started
执行工具
emit tool_call_finished 或 tool_call_failed
返回 ToolResult
```

伪代码：

```ts
async function callTool(request: ToolCallRequest): Promise<ToolResult> {
  const tool = toolRegistry.get(request.toolName);
  if (!tool) throw new Error("Unknown tool");

  await permissionManager.assertAgentCanUseTool(request.agentId, tool.name);
  validateInput(tool.inputSchema, request.input);

  if (tool.requiresApproval) {
    await permissionManager.requestUserApproval(request, tool);
  }

  await eventBus.emitToolCallStarted(request);

  try {
    const result = await tool.handler(request.input, request.context);
    await eventBus.emitToolCallFinished(request, result);
    return result;
  } catch (error) {
    await eventBus.emitToolCallFailed(request, error);
    throw error;
  }
}
```

## 6. 工具风险等级

### low

无需用户确认。

示例：

```text
文本摘要
本地只读配置查询
mock search
读取应用内部 session 数据
```

### medium

默认可执行，但用户可在设置中要求确认。

示例：

```text
网页搜索
读取用户指定目录内文件
写入项目 workspace 下文件
```

### high

必须用户确认。

示例：

```text
运行终端命令
删除文件
覆盖文件
访问任意本地路径
向外部服务上传文件
浏览器登录态操作
```

## 7. MVP 工具列表

### 7.1 web_search，P0 mock 实现

P0 暴露给 Agent 的工具名统一为 `web_search`，但实现使用 mock provider，便于开发 UI。P1 在不改 Agent 配置和事件协议的前提下，将同名工具切换为真实搜索 provider。

输入：

```json
{
  "query": "string"
}
```

输出：

```json
{
  "items": [
    {
      "title": "string",
      "url": "string",
      "summary": "string"
    }
  ]
}
```

风险：low。

---

### 7.2 web_search 真实 provider，P1

真实联网搜索。

风险：medium。

要求：

```text
记录 query
记录来源 URL
不要绕过网站访问限制
结果进入 tool_call_finished payload 时要做摘要
```

---

### 7.3 file_read，P1

读取用户授权 workspace 内文件。

风险：medium。

限制：

```text
只能读 workspacePath 内文件
不允许读 ~/.ssh、系统密钥、浏览器数据等敏感路径
超大文件要截断或摘要，并明确提示
```

---

### 7.4 file_write，P2

写入 workspace 文件。

风险：high。

要求：

```text
写入前展示 diff
用户确认后写入
禁止覆盖 workspace 外文件
```

---

### 7.5 terminal，P2

运行终端命令。

风险：high。

要求：

```text
必须用户确认
禁止默认执行 rm -rf、sudo、chmod 777 等高危命令
命令输出实时事件化
设置超时
```

---

### 7.6 browser，P2

浏览器操作。

风险：high。

要求：

```text
明确显示当前 URL
敏感站点操作前确认
不自动提交表单、付款、发消息
截图和点击轨迹进入事件流
```

## 8. Agent 工具白名单

每个 Agent 配置 tools：

```json
{
  "id": "agent_researcher",
  "tools": ["web_search"]
}
```

P1 增加 `web_fetch` 时，再把它加入 ResearchAgent 白名单。

ToolRuntime 必须检查：

```text
Agent 是否启用该工具
工具是否存在
工具风险是否允许
是否需要用户确认
```

## 9. 用户授权流程

高风险工具触发：

```text
tool_permission_requested event
↓
UI 右侧显示工具、输入、风险说明
↓
用户批准或拒绝
↓
tool_permission_granted / denied event
↓
ToolRuntime 继续或中止
```

事件示例：

```json
{
  "type": "tool_permission_requested",
  "summary": "编码智能体请求运行终端命令",
  "payload": {
    "toolName": "terminal",
    "input": {
      "command": "npm test"
    },
    "riskLevel": "high",
    "reason": "需要运行测试验证代码"
  }
}
```

## 10. API Key 安全

### 10.1 存储原则

```text
不要把 API Key 明文存 SQLite
不要把 API Key 发到前端
不要把 API Key 写入日志或事件 payload
不要把 API Key 放进 crash report
```

---

### 10.2 推荐实现

```text
macOS：Keychain
Windows：Credential Manager
Linux：Secret Service
```

SQLite 只存：

```text
api_key_ref
provider
name
keychain_service
keychain_account
```

后端调用 ProviderAdapter 时从 Keychain 读取明文。

---

## 11. 日志脱敏

事件 payload 里不得包含：

```text
API Key
Authorization header
Cookie
密码
私钥
完整本地敏感路径，必要时可模糊化
```

脱敏规则：

```text
sk-abc123... -> sk-***
Bearer xxx -> Bearer ***
/Users/name/.ssh/id_rsa -> ~/.ssh/id_rsa，且禁止读取
```

## 12. 文件系统安全

P1/P2 工具要引入 workspace 概念：

```text
每个 session 可绑定一个 workspacePath
所有文件工具默认限制在 workspacePath 内
访问 workspace 外路径必须禁止或用户确认
```

路径校验：

```ts
function assertInsideWorkspace(path: string, workspacePath: string) {
  const resolved = resolve(path);
  const workspace = resolve(workspacePath);
  if (!resolved.startsWith(workspace)) {
    throw new Error("Path outside workspace is not allowed");
  }
}
```

还要防止符号链接逃逸，实际实现需 resolve realpath。

---

## 13. 网络安全

web 工具要求：

```text
不要访问内网地址，除非用户明确允许
不要自动提交表单
不要绕过登录或反爬
不要进行批量高频请求
不要执行破坏性网络操作
```

---

## 14. 安全验收标准

```text
1. 前端拿不到 API Key 明文
2. SQLite 中没有 API Key 明文
3. 每次工具调用都有事件记录
4. Agent 不能调用未授权工具
5. 高风险工具必须用户确认
6. 工具输入输出在 Trace Console 可见
7. 日志中没有 Authorization/Cookie/API Key
8. 文件工具不能访问 workspace 外路径
9. 终端工具默认不开启
10. 工具失败有明确错误事件
```
