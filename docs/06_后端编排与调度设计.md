# Agent Workbench 后端编排与调度设计

## 1. 后端职责

后端负责所有不可交给前端的核心逻辑：

```text
1. 创建 session
2. 调用总智能体规划任务
3. 校验任务 DAG
4. 调度多个 Agent 并行执行
5. 统一调用 LLM Provider
6. 执行工具调用
7. 管理权限和 API Key
8. 生成事件并推送前端
9. 持久化 session、task、event
10. 汇总最终结果
```

## 4. Orchestrator 设计

### 4.1 职责

```text
任务前：调用总智能体生成任务计划
任务中：通常不参与调度，调度由 Scheduler 负责
任务后：收集任务结果并生成最终答案
```

---

### 4.2 规划流程

```text
Orchestrator.run(sessionId)
↓
emit planning_started
↓
读取可用 Agent 配置
↓
调用 ProviderAdapter 请求总智能体
↓
得到结构化 plan JSON
↓
校验 plan
↓
创建 tasks
↓
emit plan_created 和 task_created
↓
Scheduler.start(sessionId)
```

---

### 4.3 Plan Schema

```ts
type TaskPlan = {
  tasks: Array<{
    id: string;
    title: string;
    agentId: string;
    instruction: string;
    dependsOn: string[];
    expectedOutput: string;
  }>;
};
```

校验规则：

```text
tasks 不能为空
每个 id 唯一
agentId 必须存在
dependsOn 必须引用存在 task
不能有循环依赖
必须至少有一个无依赖 task
每个 instruction 不能为空
```

---

### 4.4 循环依赖检测

使用 DFS：

```ts
function assertAcyclic(tasks: AgentTask[]) {
  const graph = new Map(tasks.map(task => [task.id, task.dependsOn]));
  const visiting = new Set<string>();
  const visited = new Set<string>();

  function visit(id: string) {
    if (visiting.has(id)) throw new Error(`Cycle detected at ${id}`);
    if (visited.has(id)) return;

    visiting.add(id);
    for (const dep of graph.get(id) ?? []) visit(dep);
    visiting.delete(id);
    visited.add(id);
  }

  for (const task of tasks) visit(task.id);
}
```

## 5. Scheduler 设计

### 5.1 职责

```text
根据 task.dependsOn 判断可运行任务
控制最大并发
启动 AgentRuntime
监听任务完成/失败
决定下游任务是否可运行
处理暂停/取消
```

---

### 5.2 调度状态

```ts
type SchedulerState = {
  sessionId: string;
  maxConcurrency: number;
  runningTaskIds: Set<string>;
  cancelled: boolean;
  paused: boolean;
};
```

---

### 5.3 可运行任务判断

```ts
function getRunnableTasks(tasks: AgentTask[]) {
  return tasks.filter(task => {
    if (task.status !== "pending" && task.status !== "waiting") return false;

    return task.dependsOn.every(depId => {
      const dep = tasks.find(item => item.id === depId);
      return dep?.status === "success";
    });
  });
}
```

---

### 5.4 主循环

```ts
async function schedule(sessionId: string) {
  while (true) {
    if (state.cancelled) break;
    if (state.paused) await waitUntilResumed();

    const tasks = await taskRepo.listBySession(sessionId);

    if (allTasksFinished(tasks)) {
      await orchestrator.summarize(sessionId);
      break;
    }

    if (hasFailedBlockingTask(tasks)) {
      await failSession(sessionId);
      break;
    }

    const runnable = getRunnableTasks(tasks).slice(
      0,
      state.maxConcurrency - state.runningTaskIds.size
    );

    for (const task of runnable) {
      startTask(task);
    }

    await waitForTaskStateChange();
  }
}
```

实现时不要用 busy loop。任务完成时通过 Promise、事件或队列唤醒调度器。

## 6. AgentRuntime 设计

### 6.1 输入

```ts
type RunAgentTaskInput = {
  sessionId: string;
  taskId: string;
  agentId: string;
};
```

### 6.2 执行流程

```text
读取 task
读取 agent
读取依赖任务结果
emit task_started
构造 messages
调用 ProviderAdapter.streamMessage
处理 text delta
处理 tool use
收集最终输出
更新 task result
emit task_finished
```

---

### 6.3 伪代码

```ts
async function runAgentTask(input: RunAgentTaskInput) {
  const task = await taskRepo.get(input.taskId);
  const agent = await agentRepo.get(input.agentId);
  const dependencyResults = await taskRepo.getDependencyResults(task.dependsOn);

  await taskRepo.markRunning(task.id);
  await eventBus.emitTaskStarted(task, agent);

  let finalText = "";

  try {
    const stream = provider.streamMessage({
      provider: agent.provider,
      model: agent.model,
      systemPrompt: agent.systemPrompt,
      messages: buildTaskMessages(task, dependencyResults),
      tools: toolRegistry.getToolSchemas(agent.tools)
    });

    for await (const event of stream) {
      if (event.type === "text_delta") {
        finalText += event.text;
        await eventBus.emitAgentMessageDelta(task, agent, event.text);
      }

      if (event.type === "tool_call") {
        const result = await toolRuntime.call({
          sessionId: task.sessionId,
          taskId: task.id,
          agentId: agent.id,
          toolName: event.name,
          input: event.input
        });

        await provider.submitToolResult(event.toolCallId, result);
      }
    }

    await taskRepo.markSuccess(task.id, finalText);
    await eventBus.emitTaskFinished(task, agent, finalText);
  } catch (error) {
    await taskRepo.markFailed(task.id, serializeError(error));
    await eventBus.emitTaskFailed(task, agent, error);
    throw error;
  }
}
```

具体 Provider 的工具循环实现可根据 SDK 能力调整，但对外必须保持事件一致。

## 7. 并行执行策略

MVP 最大并发：3。

规则：

```text
无依赖任务可以并行
同一 Agent 默认同一时间只执行一个任务
不同 Agent 可并行执行
总智能体规划和总结通常独占运行
```

如果两个任务分配给同一个 Agent：

```text
P0：串行执行
P1：支持 Agent 多实例并发
```

---

## 8. 失败策略

### 8.1 任务失败

```text
emit task_failed
更新 task.status = failed
如果下游依赖该任务，则下游不启动
session 默认 failed
```

### 8.2 可重试错误

错误 payload 包含：

```json
{
  "retryable": true,
  "suggestedAction": "retry"
}
```

P1 支持用户点击重试。

### 8.3 工具失败

工具失败不一定导致任务失败，由 AgentRuntime 决定是否把错误反馈给模型继续处理。

MVP 简化：

```text
工具失败 -> task_failed
```

P1：工具失败返回给模型，让 Agent 选择替代方案。

---

## 9. 暂停与取消

### 9.1 暂停

MVP 暂停策略：

```text
不启动新任务
正在运行任务继续完成
session.status = paused
```

P1 支持中断运行中的 LLM stream。

### 9.2 取消

取消策略：

```text
停止启动新任务
尝试取消 running task
标记未完成任务为 cancelled
emit session_cancelled
```

## 10. 最终汇总

当所有任务 success：

```text
Orchestrator 收集所有 task.result
按任务依赖顺序整理上下文
调用总智能体生成 finalAnswer
写 sessions.final_answer
emit session_finished
```

汇总 prompt 应包含：

```text
原始用户任务
任务计划
每个 Agent 的输出
审查 Agent 的问题和建议
输出格式要求
```

## 13. 后端验收标准

```text
1. POST /sessions 可以创建 session
2. 创建 session 后立刻写 session_started event
3. Orchestrator 能生成有效 task plan
4. Scheduler 能并行启动无依赖任务
5. 每个 task_started/task_finished 都有事件
6. AgentRuntime 能流式发送 agent_message_delta
7. 工具调用能发送 started/finished 事件
8. 所有事件写入 SQLite
9. 前端能实时收到事件
10. session 完成后 sessions.final_answer 有值
```
