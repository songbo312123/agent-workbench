# Agent Workbench 事件流协议

## 1. 协议目标

事件流是 Agent Workbench 的核心。所有动态 UI、Trace Console、任务回放、状态恢复都依赖事件流。

本协议目标：

```text
1. 描述所有 Agent 行为
2. 支持实时推送到前端
3. 支持持久化到 SQLite
4. 支持从历史事件恢复 UI 状态
5. 支持未来任务回放
6. 支持调试和审计
```

---

## 2. 设计原则

### 2.1 事件只追加，不修改

事件一旦产生，只能追加，不应修改。

状态修正通过新事件表达，例如：

```text
task_failed
retry_requested
task_restarted
task_finished
```

---

### 2.2 UI 状态必须可由事件推导

前端状态应由：

```text
events[] -> reducer -> WorkbenchState
```

推导出来。

---

### 2.3 事件必须可读

事件不仅给机器用，也给用户和开发者看。因此 payload 要避免存不可理解的原始对象。

每个事件应包含：

```text
机器可处理字段
人类可读 summary
可展开 payload
```

---

## 3. 基础事件结构

```ts
type AgentWorkbenchEvent = {
  id: string;
  version: "aw.event.v1";
  type: AgentEventType;
  sessionId: string;
  taskId?: string;
  agentId?: string;
  parentEventId?: string;
  timestamp: string;
  sequence: number;
  severity: "debug" | "info" | "warning" | "error";
  summary: string;
  payload: Record<string, unknown>;
};
```

字段说明：

| 字段 | 说明 |
|---|---|
| `id` | 全局唯一事件 ID |
| `version` | 协议版本，MVP 固定 `aw.event.v1` |
| `type` | 事件类型 |
| `sessionId` | 所属 session |
| `taskId` | 可选，所属任务 |
| `agentId` | 可选，所属 Agent |
| `parentEventId` | 可选，表示事件关系 |
| `timestamp` | ISO 时间 |
| `sequence` | session 内递增序号 |
| `severity` | 日志级别 |
| `summary` | 人类可读摘要 |
| `payload` | 结构化数据 |

## 5. 事件示例

事件 payload 必须有稳定 schema。实现时建议在 `packages/shared/schemas` 中维护 Zod 或 JSON Schema，并由前端、后端、测试共用。

MVP 核心 payload schema：

```ts
type SessionStartedPayload = {
  userTask: string;
};

type PlanningStartedPayload = {
  model: string;
};

type PlanCreatedPayload = {
  tasks: Array<{
    id: string;
    title: string;
    agentId: string;
    instruction: string;
    dependsOn: string[];
    expectedOutput: string;
  }>;
};

type TaskCreatedPayload = {
  task: {
    id: string;
    title: string;
    agentId: string;
    instruction: string;
    dependsOn: string[];
    expectedOutput: string;
  };
};

type TaskStartedPayload = {
  taskTitle: string;
  instruction: string;
};

type AgentMessageDeltaPayload = {
  text: string;
};

type ToolCallStartedPayload = {
  toolCallId: string;
  toolName: string;
  input: Record<string, unknown>;
};

type ToolCallFinishedPayload = {
  toolCallId: string;
  toolName: string;
  durationMs: number;
  outputSummary: string;
  output?: unknown;
};

type TaskFinishedPayload = {
  result: string;
  outputRef?: string | null;
};

type TaskFailedPayload = {
  errorCode: string;
  errorMessage: string;
  retryable: boolean;
  suggestedAction?: "retry" | "change_input" | "configure_api_key" | "none";
};

type SessionFinishedPayload = {
  finalAnswer: string;
};
```

规则：

```text
1. 示例 JSON 必须能通过对应 schema 校验。
2. 后端 emit 前必须校验 payload。
3. 前端 reducer 不能依赖 schema 外字段。
4. payload 不允许包含 API Key、Authorization、Cookie、私钥、完整敏感路径。
5. 大内容只放摘要和 artifact/outputRef，不直接塞入 payload。
```

## 6. 状态推导规则

### 6.1 Session 状态

| 当前事件 | Session 状态 |
|---|---|
| `session_started` | planning |
| `plan_created` | running |
| `session_paused` | paused |
| `session_finished` | completed |
| `session_failed` | failed |
| `session_cancelled` | cancelled |

---

### 6.2 Task 状态

| 当前事件 | Task 状态 |
|---|---|
| `task_created` | pending 或 waiting |
| `task_waiting` | waiting |
| `task_started` | running |
| `task_progress` | running |
| `task_finished` | success |
| `task_failed` | failed |
| `task_cancelled` | cancelled |

---

### 6.3 Agent 状态

| 当前事件 | Agent 状态 |
|---|---|
| `agent_started` | running |
| `agent_status_changed` | payload.status |
| `tool_call_started` | tool_call |
| `tool_call_finished` | running |
| `task_finished` | success 或 idle，取决于是否还有任务 |
| `task_failed` | failed |

---

### 6.4 状态冲突处理

当 `tasks` 表状态和 `events` 重放结果冲突时：

```text
1. UI 状态以 events 按 sequence 重放结果为准。
2. Scheduler 状态以 tasks 表为准，但写入 tasks 必须与对应 event 在同一事务内完成。
3. 检测到冲突时，后端必须记录 warning 日志，并提供修复脚本或开发命令重新投影 tasks。
4. 禁止前端为了修正 UI 直接改 task/session 状态。
```

## 7. 前端 reducer 要求

前端 reducer 必须满足：

```text
输入同一批 events，输出相同 WorkbenchState
可以重复运行
不依赖当前时间
不直接触发副作用
```

伪代码：

```ts
function reduceEvents(events: AgentWorkbenchEvent[]): WorkbenchState {
  return events.reduce(reduceEvent, createInitialState());
}
```

---

## 8. 事件顺序与幂等

### 8.1 顺序

后端为每个 session 分配递增 `sequence`。

前端收到事件时按：

```text
sequence ASC
```

处理。

### 8.2 幂等

前端 EventStore 以 `event.id` 去重。重复事件不应导致 UI 状态错误。

## 9. 事件持久化要求

每个事件必须写入 `events` 表。

最小字段：

```text
id
session_id
task_id
agent_id
type
sequence
severity
summary
payload_json
created_at
```

详见 `04_数据模型与数据库设计.md`。

## 10. UI 映射

| 事件 | UI 影响 |
|---|---|
| `session_started` | 清空旧工作台，创建新 session |
| `planning_started` | 总智能体卡片进入 planning |
| `plan_created` | 工作台出现任务结构 |
| `task_started` | 对应 Agent 泳道出现事件块 |
| `agent_message_delta` | Agent 卡片更新最新输出 |
| `tool_call_started` | 事件块闪烁，Inspector 可看工具输入 |
| `tool_call_finished` | Inspector 显示工具结果 |
| `task_finished` | 节点/卡片变绿色 |
| `task_failed` | 节点/卡片变红色 |
| `session_finished` | 展示最终结果 |

---

## 11. 协议扩展规则

新增事件类型时必须：

```text
1. 加入事件类型枚举
2. 定义 payload schema
3. 更新前端 reducer
4. 更新 Trace Console 显示文案
5. 增加测试用例
```

不要随意把大量临时字段塞进 payload。payload 应该稳定、可解释。
